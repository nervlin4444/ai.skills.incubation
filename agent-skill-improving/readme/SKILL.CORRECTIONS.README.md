# SKILL.CORRECTIONS 解釋書

## 文件定位

| 檔案 | 讀者 | 用途 |
|------|------|------|
| SKILL.CORRECTIONS.md v2.5.0 | LLM | 執行指令集，條件反射觸發 |
| SKILL.CORRECTIONS.README.md（本檔） | 主人 | 口訣含義解釋、觸發時機參考、新經驗說明 |

LLM不讀取本檔案。本檔案僅供主人理解口訣設計與錯誤分類邏輯。

## 口訣設計原理

口訣採用「兩個字+句號」結構，目的是讓LLM經過百幾次查看後能條件反射理解，無需逐字閱讀上下文。

| 口訣 | 字面含義 | 對應錯誤類型 | 為何這樣設計 |
|------|---------|------------|------------|
| 斷。鏈。 | 鏈條斷開 | 技能載入失敗、記憶未寫入、順序錯亂、SOUL未注入 | 技能之間像鏈條一環扣一環，斷開就無法運作 |
| 偏。軌。 | 偏離軌道 | 工作流階段錯亂、未經確認覆蓋腳本、bootstrap加載時機錯誤 | 像火車偏離軌道，必須立即剎停糾正 |
| 絆。腳。 | 絆倒腳步 | 版本號錯誤、創建臨時腳本、格式違規、名稱特化、擅自改進技能 | 小石頭絆腳，不致命但會拖慢進度 |

## 三類錯誤的區分標準

### 斷鏈（P0 致命）

當技能根本無法載入，或SOUL.md未正確注入時觸發。

典型場景：
- 呼叫use_skill("agent-bootstrap")報錯「找不到技能」
- Session開始時跳過SOUL.md直接處理用戶消息
- 改進技能後沒有寫入MEMORY.md，下次session重複犯相同錯誤
- **SOUL.md v5.0未注入，導致Agent忘記肌肉記憶（v2.5.0 新增）**
- **IDENTITY.md未注入，導致Agent姿態不明（v2.5.0 新增）**

為何致命：地基不穩，後續所有工作都是無效勞動。

### 偏軌（P1 嚴重）

當技能能載入，但執行方向錯誤或加載時機錯誤時觸發。

典型場景：
- Planning階段還沒完成就開始Crafting
- Evaluating發現問題後沒有回到Crafting修正，直接進入Finishing
- 生成新腳本後未經主人確認就覆蓋舊版本
- **agent-bootstrap在對話開始時加載，而非技能完成後（v2.5.0 新增）**
- **發現技能缺陷後擅自改進，未經主人確認（v2.5.0 新增）**

為何嚴重：方向錯了，做得越快錯得越遠。

### 絆腳（P2 拖慢）

當方向正確，但細節違規時觸發。

典型場景：
- 版本號從v2.5.0改成v2.5.1，但實際是新增功能應改為v2.6.0
- 明明有conversation_append.py模塊，卻又創建write_final.py臨時腳本
- frontmatter的description用了 `>` 多行折疊，導致WorkBuddy載入失敗
- SKILL.md中出現「林總」「香港」等特定名稱，導致LLM綁定特定環境
- 生成markdown後只在對話中展示，沒有提供可下載連結
- **將agent-*相關記憶壓縮為摘要，丟失踩坑因果鏈（v2.5.0 新增）**

為何不致命：不會導致任務失敗，但會浪費時間返工。

## 新經驗說明（v2.5.0 新增）

### 1. SOUL.md 優先注入

**問題**：Agent跳過SOUL.md直接處理用戶消息，導致忘記「先啟動」「備歸檔」等肌肉記憶。

**解決**：SOUL.md v5.0是每輪對話第一個注入的檔案，建立身份內化與肌肉記憶。IDENTITY.md v5.0是第二個注入的檔案，確認姿態。禁止在讀完這兩個檔案前回應用戶。

### 2. bootstrap 加載時機

**問題**：Agent在對話開始時加載bootstrap，輸出被平台發送→Connector捕獲→回環。

**解決**：agent-bootstrap v2.5.0改為「技能完成後」加載，評估是否需要更深層技能。對話開始時只注入SOUL.md + IDENTITY.md。

### 3. 技能改進是操作手冊

**問題**：Agent發現技能缺陷後直接修改，未經主人確認，導致資產被擅自改動。

**解決**：agent-skill-improving v1.2.3改為操作手冊定位。發現缺陷→建議主人→等待確認→按手冊執行（備份→Patch→驗證→記錄）。禁止擅自改進。

### 4. 記憶禁止壓縮

**問題**：Agent將踩坑記錄壓縮為摘要，丟失完整因果鏈，下次照踩。

**解決**：SKILL.CORRECTIONS.md v2.5.0新增「記憶壓縮禁令」。必須保留完整Context / Action / Result / Correction / Lesson五段式記憶。

## 錯誤記錄存檔說明

SKILL.CORRECTIONS.md底部的「錯誤記錄存檔」表格僅供主人追溯歷史，LLM在執行檢查清單時不會讀取這些記錄。這是刻意的設計——避免歷史噪音干擾LLM的當前判斷。

若主人需要LLM參考歷史錯誤，應改為讀取MEMORY.md或CONVERSATION.md。

## 信息噪音事後觀察

### 觀察目的

本文件經過「信息噪音比例」審查，確保對目標讀者（主人）無關的內容被完全排除。

### 噪音控制策略

| 策略 | 實施方式 | 效果 |
|------|---------|------|
| **讀者分流** | 用目錄區分讀者（assets/給LLM，readme/給主人），而非在同一文件內混合 | LLM執行細節不會出現在本文件中 |
| **內容聚焦** | 只解釋「為何這樣設計」和「什麼時候觸發」，不復述LLM的執行步驟 | 主人不需要知道LLM如何逐條勾選checkbox |
| **口訣不解釋語法** | 只說明口訣的字面含義和對應錯誤類型，不說明LLM如何條件反射 | 語法細節屬於LLM執行版，對人類是噪音 |
| **歷史記錄隔離** | 錯誤記錄存檔只放在assets版底部，readme版不保留 | 避免歷史表格佔用主人注意力 |
| **版本歷史精簡** | 只保留版本號、日期、一句話變更摘要 | 不展開詳細修改對照 |
